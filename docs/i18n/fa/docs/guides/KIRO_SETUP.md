# راهنمای راه‌اندازی Kiro

این راهنما افزودن حساب‌های Kiro (دستیار کدنویسی مبتنی بر AWS) به RouteChi را پوشش می‌دهد،
با تمرکز بر اجرای هم‌زمان چند حساب بدون تداخل نشست.

---

## پیش‌زمینه: چرا حساب‌های Kiro می‌توانند تداخل ایجاد کنند

بک‌اند Kiro از ثبت‌نام‌های AWS SSO OIDC client برای پیگیری نشست‌های فعال استفاده می‌کند.
محدودیت حیاتی این است: **هر ثبت‌نام OIDC client تنها از یک نشست فعال در هر زمان پشتیبانی می‌کند**.
هنگامی که دستگاه یا کاربر دوم با همان کلاینت ثبت‌شده احراز هویت می‌کند، بک‌اند refresh token
حساب اول را نامعتبر می‌سازد.

این همان مکانیزمی است که هنگام اجرای `kiro-cli login` روی ماشینی که حساب Kiro دیگری از قبل
در آن وارد شده است مشکلات ایجاد می‌کند — ورود جدید، توکن حساب اول را باطل می‌کند.

---

## نحوهٔ حل این مسئله توسط RouteChi (v3.8.0 به بعد)

از نسخهٔ v3.8.0، RouteChi در هر واردسازی اتصال Kiro تابع `registerClient()` (AWS SSO OIDC)
را فراخوانی می‌کند. این کار به هر اتصال RouteChi یک ثبت‌نام OIDC client اختصاصی می‌دهد.
چون هر ثبت‌نام کلاینت مستقل است، بازنشانی یا احراز هویت مجدد یک حساب روی refresh token
هیچ حساب دیگری تأثیری ندارد.

این ایزوله‌سازی به روش‌های واردسازی مبتنی بر refresh-token اعمال می‌شود و احراز هویت
API-key اصلاً از نشست‌های بازنشانی OIDC عبور نمی‌کند:

| روش واردسازی                                    | وضعیت ایزوله‌سازی                                                                                  |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| AWS Builder ID / IDC device-code flow           | از زمان معرفی device-code flow ایزوله بوده است                                                     |
| **Import Token** (جای‌گذاری دستی refresh token) | از v3.8.0 ایزوله شده است                                                                          |
| **ورود اجتماعی Google / GitHub**                | از v3.8.0 ایزوله شده است                                                                          |
| **Auto-Import** (SQLite مربوط به kiro-cli)      | از v3.8.0 ایزوله شده است (مسیر SQLite از قبل ایزوله بود؛ حالا SSO-cache fallback هم ایزوله شده است) |
| **API Key** (کلید طولانی‌عمر CodeWhisperer)     | بدون نشست بازنشانی؛ کلید اعتبارسنجی و به‌عنوان credential bearer ذخیره می‌شود                     |

---

## یادداشت مهاجرت برای اتصالات ایجاد‌شده پیش از v3.8.0

اتصالاتی که پیش از v3.8.0 وارد شده‌اند دارای ثبت‌نام اختصاصی OIDC client در
`providerSpecificData` نیستند. این اتصالات همچنان کار می‌کنند اما از نقطهٔ پایانی
مشترک social-auth refresh استفاده می‌کنند، به این معنی که دو اتصال از این نوع همچنان
می‌توانند یکدیگر را نامعتبر کنند.

**برای بهره‌گیری از ایزوله‌سازی:** اتصال قدیمی را از **Dashboard → Providers** حذف کنید
و با استفاده از هر یک از جریان‌های واردسازی پشتیبانی‌شده دوباره وارد کنید. تمام اتصالات
جدید به‌طور خودکار ثبت‌نام کلاینت اختصاصی خود را دریافت می‌کنند.

## افزودن دو حساب Kiro به‌صورت کنار هم

### پیش‌نیازها

- RouteChi نسخهٔ v3.8.0 یا جدیدتر.
- یک حساب کاری Kiro (ایمیل + گذرواژه، ورود با Google یا GitHub).
- در صورت تمایل، یک حساب Kiro دوم.

### گام ۱: وارد کردن حساب اول

1. باز کنید **Dashboard → Providers → Add Provider → Kiro**.
2. یکی از موارد زیر را انتخاب کنید:
   - **Import Token** — یک refresh token که با `aorAAAAAG` شروع می‌شود جای‌گذاری کنید.
   - **API Key** — یک کلید API طولانی‌عمر Kiro / CodeWhisperer جای‌گذاری کنید.
   - **ورود با Google / GitHub** — جریان OAuth را در مرورگر تکمیل کنید.
   - **Auto-Import** — روی دکمه کلیک کنید؛ RouteChi credentialها را از پایگاه دادهٔ
     محلی kiro-cli یا `~/.aws/sso/cache` می‌خواند.
3. اتصال ذخیره می‌شود. جریان‌های refresh-token به‌طور خودکار یک OIDC client اختصاصی
   ثبت می‌کنند. جریان‌های API-key کلید را با AWS اعتبارسنجی می‌کنند و refresh token ذخیره نمی‌کنند.

### گام ۲: وارد کردن حساب دوم

گام ۱ را برای حساب دوم تکرار کنید. چون هر واردسازی یک ثبت‌نام OIDC client جداگانه ایجاد
می‌کند، دو اتصال کاملاً ایزوله هستند.

### گام ۳: بررسی فعال بودن هر دو اتصال

1. **Dashboard → Providers** — هر دو اتصال Kiro باید وضعیت **Active** را نشان دهند.
2. **Dashboard → Health** — هر دو اتصال باید بررسی سلامت توکن را با موفقیت گذرانده باشند.

### گام ۴: استفاده از یک کامبو برای مسیریابی میان حساب‌ها

یک کامبو با هر دو اتصال به‌عنوان هدف بسازید تا بین آن‌ها load-balance یا fallback کنید:

```
kiro/kiro-dev → kiro/kiro-pro
```

برای پیکربندی کامبو به [FEATURES.md](./FEATURES.md) و مستندات مسیریابی مراجعه کنید.

## کاربران سازمانی / IDC

برای حساب‌های AWS IAM Identity Center (IDC) از جریان **AWS Builder ID / IDC device-code**
از **Dashboard → Providers → Kiro → Device Code** استفاده کنید. جریان device-code همیشه
کاملاً ایزوله بوده است. برای این اتصالات نیازی به واردسازی مجدد نیست.

کاربران سازمانی که در منطقهٔ AWS غیرپیش‌فرض فعالیت می‌کنند می‌توانند هنگام واردسازی
از طریق API Import Token، منطقه را مشخص کنند:

```bash
curl -X POST http://localhost:20128/api/oauth/kiro/import \
  -H "Content-Type: application/json" \
  -d '{"refreshToken": "aorAAAAAG...", "region": "eu-west-1"}'
```

فیلد `region` هنگام حذف شدن به `us-east-1` پیش‌فرض است.

## جریان واردسازی API-Key

احراز هویت API-key برای credentialهای bearer طولانی‌عمر Kiro / AWS CodeWhisperer است.
این روش از بازنشانی OAuth استفاده نمی‌کند، بنابراین از ابطال نشست مشترک OIDC جلوگیری می‌کند.

### داشبورد

1. باز کنید **Dashboard -> Providers -> Kiro**.
2. **API Key** را انتخاب کنید.
3. کلید API و منطقهٔ اختیاری AWS را جای‌گذاری کنید (`us-east-1` به‌صورت پیش‌فرض).
4. RouteChi کلید را اعتبارسنجی و اتصال را ذخیره می‌کند.

### API

```bash
curl -X POST http://localhost:20128/api/oauth/kiro/api-key \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "kiro_or_codewhisperer_key", "region": "us-east-1"}'
```

### قرارداد داخلی

مسیر API کلید را با فراخوانی `KiroService.validateApiKey()` اعتبارسنجی می‌کند که از
`ListAvailableProfiles` در برابر نقطهٔ پایانی CodeWhisperer/Amazon Q منطبق با منطقه
استفاده می‌کند و یک `profileArn` باز می‌گرداند.

اتصال ذخیره‌شده از این ساختار استفاده می‌کند:

```json
{
  "authType": "apikey",
  "providerSpecificData": {
    "authMethod": "api_key",
    "region": "us-east-1",
    "profileArn": "arn:aws:codewhisperer:..."
  }
}
```

در زمان اجرا، `KiroExecutor.buildHeaders()` کلید را به‌صورت
`Authorization: Bearer <key>` ارسال و `tokentype: API_KEY` را اضافه می‌کند. فراخوانی‌های
سهمیه/profile از همین نشانگر استفاده می‌کنند تا AWS bearer را به‌جای توکن OIDC یا
اجتماعی، به‌عنوان یک کلید API طولانی‌عمر در نظر بگیرد.

## انقضای OIDC Client

کلاینت‌های public مربوط به AWS SSO OIDC معمولاً پس از ۹۰ روز منقضی می‌شوند
(`clientSecretExpiresAt`). RouteChi این مهر زمانی را برای قابلیت مشاهده در
`providerSpecificData` ذخیره می‌کند. اگر اتصالی پس از ~۹۰ روز بازنشانی را متوقف کرد،
اتصال را دوباره وارد کنید تا یک ثبت‌نام تازه OIDC client دریافت کنید. ثبت‌نام مجدد
خودکار هنگام انقضا به‌عنوان بهبود آتی پیگیری می‌شود.

اتصالات API-key انقضای OIDC client ندارند چون از AWS SSO OIDC بازنشانی نمی‌شوند.

## رفع اشکال

### حساب دوم دائماً خارج می‌شود

- هر دو اتصال را در **Dashboard → Providers** بررسی و تأیید کنید که هر کدام در JSON خام
  خود (قابل مشاهده از طریق آیکن اطلاعات) یک `clientId` غیرتهی نشان می‌دهند. اگر هر کدام
  `clientId` نداشت، قبل از v3.8.0 وارد شده است — آن را دوباره وارد کنید.

### واردسازی با خطای "Token validation failed" شکست می‌خورد

- مطمئن شوید refresh token با `aorAAAAAG` شروع می‌شود.
- مطمئن شوید RouteChi می‌تواند به `https://oidc.us-east-1.amazonaws.com` (یا منطقهٔ
  پیکربندی‌شده) برسد. اگر پشت پراکسی سازمانی هستید، یک پراکسی در سطح ارائه‌دهنده در
  **Dashboard → Settings → Proxies** تنظیم کنید.

### شکست واردسازی API-key

- تأیید کنید که کلید یک کلید API مربوط به Kiro / CodeWhisperer است، نه یک refresh token.
- تأیید کنید که منطقهٔ AWS با کلید/حساب مطابقت دارد. `us-east-1` پیش‌فرض است.
- کلید باید بتواند `ListAvailableProfiles` را فراخوانی کند؛ در غیر این صورت RouteChi
  نمی‌تواند `profileArn` مورد نیاز را بازیابی کند.

برای سایر مشکلات، به [TROUBLESHOOTING.md](./TROUBLESHOOTING.md) اصلی مراجعه کنید.
