## خط لولهٔ ترجمه (پیشنهادی — v3.8.0)

RouteChi از یک مترجم افزایشی مبتنی بر hash برای مستندات استفاده می‌کند، که توسط یک نقطهٔ پایانی LLM سازگار با OpenAI (معمولاً `cx/gpt-5.4-mini` از طریق
RouteChi Cloud) پشتیبانی می‌شود:

```bash
# اجرای ترجمه‌ها (افزایشی — فقط منابع تغییرکرده را دست می‌زند)
npm run i18n:run

# محدود کردن به یک locale
npm run i18n:run -- --locale=pt-BR

# فایل‌های خاص (جدا‌شده با کاما، مسیرهای نسبت به repo)
npm run i18n:run -- --files=CLAUDE.md,docs/architecture/ARCHITECTURE.md

# اجبار به ترجمهٔ همه (پرهزینه)
npm run i18n:run -- --force

# پیش‌نمایش آنچه رخ می‌دهد (بدون فراخوانی API، بدون نوشتن)
npm run i18n:run:dry

# دروازهٔ CI — در صورت انحراف وضعیت، غیر صفر خارج می‌شود
npm run i18n:check
```

**منبع حقیقت.** `config/i18n.json` هر locale (UI + مستندات) به‌علاوهٔ مجموعهٔ RTL و کدهای `docsExcluded` را فهرست می‌کند. پیکربندی runtime در
`src/i18n/config.ts` یک آداپتور نازک روی آن JSON است.

**Backend.** از طریق env پیکربندی می‌شود (در `.env` تنظیم شود، هرگز commit نشود):

| متغیر                            | هدف                                 |
| ----------------------------------- | --------------------------------------- |
| `OMNIROUTE_TRANSLATION_API_URL`     | URL پایهٔ سازگار با OpenAI، مثلاً `…/v1` |
| `OMNIROUTE_TRANSLATION_API_KEY`     | bearer token (از لاگ‌ها دور نگه داشته می‌شود)         |
| `OMNIROUTE_TRANSLATION_MODEL`       | شناسهٔ مدل، مثلاً `cx/gpt-5.4-mini`        |
| `OMNIROUTE_TRANSLATION_TIMEOUT_MS`  | اختیاری، پیش‌فرض `60000`               |
| `OMNIROUTE_TRANSLATION_CONCURRENCY` | اختیاری، پیش‌فرض `4`                   |

**ردیابی وضعیت.** `.i18n-state.json` (commit می‌شود) هش‌های SHA-256 به‌ازای هر
منبع + به‌ازای هر locale را نگه می‌دارد. تشخیص انحراف خودکار و قطعی است — بدون فراخوانی API در `i18n:check`.

**شکل خروجی.** هر فایل ترجمه‌شده یک خط `# <heading>
(<native>)` در سطح بالا، یک نوار `🌐 Languages: …`، یک جداکنندهٔ `---`، و بدنهٔ ترجمه‌شده دریافت می‌کند. این چیدمان با آنچه `scripts/check/check-docs-sync.mjs`
برای `llm.txt` و mirrorهای `CHANGELOG.md` اعمال می‌کند مطابقت دارد.

### اسکریپت‌های legacy (منسوخ‌شده)

اسکریپت پایتون قدیمی (`scripts/i18n/i18n_autotranslate.py`) و مولد مبتنی بر Google-Translate (`scripts/i18n/generate-multilang.mjs`)
هنوز با یک بنر منسوخ‌شدن وجود دارند. آن‌ها در v3.10 حذف خواهند شد. حالت‌های `messages` و `readme` در `generate-multilang.mjs` (رشته‌های UI + variants ریشهٔ
README) هنوز توسط خط لولهٔ جدید مدیریت نمی‌شوند و همچنان استفاده می‌شوند.

### منبع حقیقت

- **رشته‌های UI**: `src/i18n/messages/en.json` (منبع انگلیسی، حدود ۲۸۰۰ کلید)

- **فریم‌ورک**: `next-intl` با تشخیص locale مبتنی بر cookie
- **پیکربندی**: `src/i18n/config.ts` — همهٔ ۳۰ locale، نام زبان‌ها، پرچم‌ها را تعریف می‌کند

### جریان runtime

1. کاربر زبان را انتخاب می‌کند → cookie `NEXT_LOCALE` تنظیم می‌شود
2. `src/i18n/request.ts` locale را حل می‌کند: cookie → هدر `Accept-Language` → fallback `en`
3. import پویا `messages/{locale}.json` را بارگذاری می‌کند
4. کامپوننت‌ها از `useTranslations("namespace")` و `t("key")` استفاده می‌کنند

## افزودن یک زبان جدید

### ۱. ثبت locale

`src/i18n/config.ts` را ویرایش کنید:

```ts
// Add to LOCALES array
"xx",
// Add to LANGUAGES array
{ code: "xx", label: "XX", name: "Language Name", flag: "🏳️" },
```

### ۲. افزودن به مولد

`scripts/i18n/generate-multilang.mjs` را ویرایش کنید — یک ورودی به `LOCALE_SPECS` اضافه کنید:

```js
{
  code: "xx",
  googleTl: "xx",
  label: "XX",
  flag: "🏳️",
  languageName: "Language Name",
  readmeName: "Language Name",
  docsName: "Language Name",
},
```

### ۳. تولید ترجمهٔ اولیه

```bash
node scripts/i18n/generate-multilang.mjs messages
```

این `src/i18n/messages/xx.json` را از `en.json` از طریق Google Translate خودکار ترجمه می‌کند.

### ۴. بازبینی و اصلاح ترجمه‌های خودکار

ترجمه‌های خودکار یک نقطهٔ شروع هستند. به‌صورت دستی برای موارد زیر بازبینی کنید:

- دقت فنی
- اصطلاحات مناسب با زمینه
- مدیریت صحیح placeholderها (`{count}`, `{value}` و غیره)

### ۵. اعتبارسنجی

```bash
python3 scripts/i18n/validate_translation.py quick -l xx
python3 scripts/i18n/validate_translation.py diff common -l xx
```

### ۶. تولید مستندات ترجمه‌شده

```bash
node scripts/i18n/generate-multilang.mjs docs
```

## خط لولهٔ ترجمهٔ خودکار

### generate-multilang.mjs (Google Translate)

**موتور اصلی ترجمهٔ خودکار** — از API رایگان Google Translate برای تولید ترجمه‌ها برای رشته‌های UI، READMEها و مستندات استفاده می‌کند.

```bash
node scripts/i18n/generate-multilang.mjs [messages|readme|docs|all]
```

| حالت       | کاری که می‌کند                                                                  |
| ---------- | ----------------------------------------------------------------------------- |
| `messages` | کلیدهای مفقود در `src/i18n/messages/{locale}.json` را از `en.json` ترجمه می‌کند   |
| `readme`   | `README.md` را به همهٔ localeها به‌عنوان `README.{code}.md` در ریشهٔ پروژه ترجمه می‌کند |
| `docs`     | `DOC_SOURCE_FILES` را به `docs/i18n/{locale}/{docName}` ترجمه می‌کند             |
| `all`      | هر سه حالت را اجرا می‌کند                                                          |

**ویژگی‌ها:**

- **حفاظت متن**: blockهای کد (` ``` `)، inline code (`` ` ``)، پیوند/تصاویر markdown (`[text](url)`)، تگ‌های HTML، جداول و placeholderهای ICU (`{count}`, `{value}`, `{total}` و غیره) پیش از ترجمه mask و سپس restore می‌شوند
- ** batching تکه‌تکه**: چند رشته را با جداکننده‌های `__OMNIROUTE_I18N_SEPARATOR__` برای کمینه‌سازی فراخوانی‌های API ترکیب می‌کند (حداکثر ۱۸۰۰ نویسه به ازای درخواست)
- **کش درون‌حافظه‌ای**: از فراخوانی‌های API اضافی برای رشته‌های تکراری درون یک نشست جلوگیری می‌کند
- **منطق retry**: backoff نمایی (تا ۵ تلاش با تأخیر ۳۰۰ms × شماره تلاش) برای خطاهای 429/5xx
- **timeout**: ۲۰ ثانیه به ازای درخواست
- **رد کردن فایل موجود**: اگر فایل هدف از قبل وجود داشته باشد، بازنویسی نمی‌شود

**رفتارهای مهم:**

- `docs/i18n/README.md` در هر اجرا **بازتولید** می‌شود — این یک نمایهٔ خودکار همهٔ مستندات است
- فایل‌های ریشهٔ `README.{code}.md` تنها در صورت عدم وجود ایجاد می‌شوند (localeهای داخل `EXISTING_README_CODES` را رد می‌کند)
- نوارهای زبان (`🌐 **Languages:** ...`) به‌صورت خودکار در همهٔ مستندات ترجمه‌شده درج/به‌روزرسانی می‌شوند

### i18n_autotranslate.py (مبتنی بر LLM)

**مترجم ثانویه** — از هر API LLM سازگار با OpenAI (از جمله خود RouteChi) برای ترجمهٔ فایل‌های markdown موجود `docs/i18n/` استفاده می‌کند. بهترین برای صیقل دادن یا ترجمهٔ مجدد مستندات با کیفیت بهتر از Google Translate.

```bash
python3 scripts/i18n/i18n_autotranslate.py \
  --api-url http://localhost:20128/v1 \
  --api-key sk-your-key \
  --model gpt-4o
```

**ویژگی‌ها:**

- فایل‌های markdown `docs/i18n/` را برای پاراگراف‌های انگلیسی اسکن می‌کند
- blockهای کد، جداول و محتوای از قبل ترجمه‌شده را رد می‌کند
- پاراگراف‌ها را با system prompt ترجمهٔ فنی به LLM ارسال می‌کند
- از همهٔ ۴۳ زبان پشتیبانی می‌کند

## i18n در CLI

CLI `omniroute` لایهٔ i18n جداگانه‌ای از داشبورد Next.js دارد.

### نحوه کار

- هر رشتهٔ رو‌به‌کاربر در دستورات CLI از طریق `t("module.key", vars)` از `bin/cli/i18n.mjs` عبور می‌کند.
- کاتالوگ‌ها فایل‌های JSON در `bin/cli/locales/` هستند — ۴۳ کاتالوگ از جعبه عرضه می‌شوند.
- locale برای هر کلید مفقود به `en` بازمی‌گردد، بنابراین ترجمه‌های جزئی معتبر هستند.
- منبع حقیقت برای localeهای در دسترس `config/i18n.json` است (با داشبورد مشترک است).

### انتخاب locale

ترتیب تشخیص (اولین تطابق برنده):

| اولویت | منبع                   | مثال                                 |
| -------- | ------------------------ | --------------------------------------- |
| ۱        | پرچم `--lang`            | `routechi --lang de status`            |
| ۲        | متغیر محیطی `OMNIROUTE_LANG` | `OMNIROUTE_LANG=ja omniroute providers` |
| ۳        | متغیر سیستمی `LC_ALL`      | تشخیص خودکار از locale ترمینال      |
| ۴        | متغیر سیستمی `LC_MESSAGES` | تشخیص خودکار از locale ترمینال      |
| ۵        | متغیر سیستمی `LANG`        | تشخیص خودکار از locale ترمینال      |
| ۶        | Fallback                 | `en`                                    |

کدهای locale با underscore (`pt_BR`) به‌صورت نرمال به فرمت خط تیره (`pt-BR`) تبدیل می‌شوند.
کدهای locale در برابر `/^[a-zA-Z0-9-]+$/` اعتبارسنجی می‌شوند — path traversal رد می‌شود.

### ذخیرهٔ ترجیح زبان

```bash
# تنظیم زبان و ذخیره در ~/.omniroute/.env (بین نشست‌ها ماندگار است)
routechi config lang set pt-BR

# مشاهدهٔ زبان فعلی
routechi config lang get

# فهرست همهٔ ۴۲ زبان در دسترس
routechi config lang list

# خروجی JSON
routechi config lang list --output json
```

ترجیح ذخیره‌شده به‌صورت اتمی در `~/.omniroute/.env` نوشته می‌شود و توسط bootstrap پیش از
اجرای هر دستور بارگذاری می‌شود.

### بازنویسی یک‌باره

```bash
# بازنویسی فقط برای یک دستور (ماندگار نمی‌شود)
routechi --lang de providers list
```

نکته: پرچم `--lang` در فایل env نمی‌نویسد — تنها بر فراخوانی فعلی اثر می‌گذارد.
برای ماندگاری از `config lang set` استفاده کنید.

### localeهای در دسترس

۴۳ فایل locale در `bin/cli/locales/` عرضه می‌شوند. ترجمه‌های کامل: `en`، `pt-BR`.
فقط scaffold (همهٔ کلیدها به `en` بازمی‌گردند): `bn`, `gu`, `he`, `in`, `mr`, `ms`, `phi`, `sw`, `ta`, `te`, `ur`.
همهٔ ۲۹ locale دیگر دارای کلیدهای `common` + `program` ترجمه‌شده هستند.

### افزودن یک locale جدید CLI

1. ورودی locale را به `config/i18n.json` اضافه کنید.
2. `node bin/cli/scripts/generate-locales.mjs` را اجرا کنید — فایل locale را ایجاد می‌کند.
3. کلیدها را ترجمه کنید (یا برای scaffold en-fallback خالی `{}` بگذارید).
4. PRها باید رشته‌ها را به `en.json` و `pt-BR.json` اضافه کنند؛ سایر فایل‌ها best-effort هستند.

## اعتبارسنجی و QA

### validate_translation.py

**اعتبارسنج‌کنندهٔ ترجمه** — هر JSON locale را با `en.json` مقایسه می‌کند و موارد را گزارش می‌دهد.

```bash
# بررسی سریع (فقط شمارش)
python3 scripts/i18n/validate_translation.py quick -l cs
# خروجی:
# Missing: 0
# Untranslated: 0
# Ignored (UNTRANSLATABLE_KEYS): 236

# تفصیل diff به‌تفکیک دسته
python3 scripts/i18n/validate_translation.py diff common -l cs
python3 scripts/i18n/validate_translation.py diff settings -l cs

# خروجی به CSV
python3 scripts/i18n/validate_translation.py csv -l cs > report.csv

# خروجی به Markdown
python3 scripts/i18n/validate_translation.py md -l cs > report.md

# گزارش کامل (پیش‌فرض)
python3 scripts/i18n/validate_translation.py -l cs
```

**تشخیص می‌دهد:**

- **کلیدهای مفقود** — کلیدهایی که در `en.json` هستند اما در فایل locale نه
- **کلیدهای اضافی** — کلیدهایی که در فایل locale هستند اما در `en.json` نه
- **کلیدهای ترجمه‌نشده** — کلیدهایی که مقدار locale با منبع انگلیسی برابر است (به‌جز allowlist)
- **عدم تطابق placeholder** — placeholderهای ICU که بین منبع و ترجمه مطابقت ندارند

**کدهای خروجی:**

| کد | معنا                      |
| ---- | ---------------------------- |
| 0    | OK                           |
| 1    | خطای عمومی                |
| 2    | رشته‌های مفقود (خطای سخت) |
| 3    | هشدار ترجمه‌نشده (نرم)  |

**محیط:** `TRANSLATION_LANG=cs` را تنظیم یا از پرچم `-l cs` استفاده کنید.

### check_translations.py

**بررسی‌کنندهٔ کلید کد‌به‌JSON** — `src/**/*.tsx` و `src/**/*.ts` را برای فراخوانی‌های `useTranslations()` اسکن می‌کند و تأیید می‌کند که همهٔ کلیدهای ارجاع‌شده در `en.json` موجودند.

```bash
# بررسی پایه
python3 scripts/i18n/check_translations.py

# خروجی verbose
python3 scripts/i18n/check_translations.py --verbose

# اصلاح خودکار (کلیدهای مفقود را به en.json اضافه می‌کند)
python3 scripts/i18n/check_translations.py --fix
```

### generate-qa-checklist.mjs

**QA تحلیل استاتیک** — فایل‌های صفحهٔ Next.js را برای معیارهای ریسک i18n اسکن می‌کند و یک گزارش Markdown تولید می‌کند.

```bash
node scripts/i18n/generate-qa-checklist.mjs
```

**بررسی می‌کند:**

- استفاده از کلاس عرض ثابت (ریسک سرریز)
- کلاس‌های جهت‌دار left/right (ریسک RTL)
- الگوهای مستعد clipping
- برابری locale (کلیدهای مفقود/اضافی در برابر `en.json`)
- نوارهای انتخابگر زبان README در localeهای با اولویت (`es`, `fr`, `de`, `ja`, `ar`)

**خروجی:** `docs/reports/i18n-qa-checklist-{date}.md`

### run-visual-qa.mjs

**QA بصری با Playwright** — از همهٔ مسیرهای داشبورد در چندین locale و viewport اسکرین‌شات می‌گیرد و سپس سلامت صفحه را ارزیابی می‌کند.

```bash
# پیش‌فرض: es, fr, de, ja, ar روی localhost:20128
node scripts/i18n/run-visual-qa.mjs

# URL پایه و localeهای سفارشی
QA_BASE_URL=http://staging.example.com QA_LOCALES=de,fr node scripts/i18n/run-visual-qa.mjs

# مسیرهای سفارشی
QA_ROUTES=/dashboard/settings,/dashboard/providers node scripts/i18n/run-visual-qa.mjs
```

**تشخیص می‌دهد:**

- سرریز متن
- clipping عنصر
- عدم تطابق چیدمان RTL

**خروجی:** `docs/reports/i18n-visual-qa-{date}.md` + گزارش JSON

## مدیریت کلیدهای غیرقابل‌ترجمه

### untranslatable-keys.json

**فایل:** `scripts/i18n/untranslatable-keys.json`

Allowlist کلیدهایی که باید با منبع انگلیسی یکسان بمانند. توسط `validate_translation.py` برای جلوگیری از هشدارهای مثبت کاذب «ترجمه‌نشده» استفاده می‌شود.

```json
{
  "description": "Keys that should remain untranslated...",
  "keys": [
    "common.model",
    "common.oauth",
    "health.cpu",
    ...
  ]
}
```

**چه چیزی به اینجا تعلق دارد:**

- نام‌های برند/محصول: `landing.brandName`, `common.social-github`
- اصطلاحات فنی/سرنام‌ها: `health.cpu`, `mcpDashboard.pid`, `settings.ai`
- رشته‌های ICU/قالب: `apiManager.modelsCount`, `health.millisecondsShort`
- مقادیر placeholder: `providers.openaiBaseUrlPlaceholder`, `cliTools.baseUrlPlaceholder`
- نام‌های پروتکل: `common.http`, `common.oauth`, `providers.oauth2Label`
- بخش‌های ناوبری: `sidebar.primarySection`, `sidebar.cliSection`

**برای افزودن یک کلید:** آرایهٔ `keys` در `scripts/i18n/untranslatable-keys.json` را ویرایش کرده و اعتبارسنجی را مجدداً اجرا کنید.

## یکپارچه‌سازی CI

### GitHub Actions (`.github/workflows/ci.yml`)

خط لولهٔ CI همهٔ localeها را در هر push و PR اعتبارسنجی می‌کند:

1. **job `i18n-matrix`** — به‌صورت پویا همهٔ فایل‌های locale را کشف می‌کند (به‌جز `en.json`)
2. **job `i18n`** — `validate_translation.py quick -l '<lang>'` را به‌ازای هر locale به‌موازات اجرا می‌کند
3. **job `ci-summary`** — نتایج را در یک خلاصهٔ داشبورد جمع می‌کند

```yaml
# i18n-matrix: discovers languages
LANGS=$(ls src/i18n/messages/*.json | xargs -n1 basename | sed 's/.json$//' | grep -v '^en$')

# i18n: validates each language
python3 scripts/i18n/validate_translation.py quick -l '${{ matrix.lang }}'
```

**خروجی داشبورد:**

```
## 🌍 Translations
| Metric | Value |
|--------|------|
| Languages checked | 30 |
| Total untranslated | 0 |

✅ All translations complete
```

## بهترین رویه‌ها

### هنگام ویرایش ترجمه‌ها

1. **همیشه ابتدا `en.json` را ویرایش کنید** — این منبع حقیقت است
2. **`generate-multilang.mjs messages` را اجرا کنید** تا کلیدهای جدید به همهٔ localeها منتقل شوند
3. **ترجمه‌های خودکار را بازبینی کنید** — Google Translate یک نقطهٔ شروع است، نه نهایی
4. **پیش از commit اعتبارسنجی کنید** — `python3 scripts/i18n/validate_translation.py quick -l <lang>`
5. **`untranslatable-keys.json` را به‌روزرسانی کنید** اگر کلیدی باید به انگلیسی باقی بماند

### ایمنی placeholder

- placeholderهای ICU (`{count}`, `{value}`, `{total}`, `{seconds}`) باید دقیقاً حفظ شوند
- قالب‌های جمع (`{count, plural, one {# model} other {# models}}`) باید ساختار خود را حفظ کنند
- اعتبارسنج‌کننده عدم تطابق placeholder را به‌صورت خودکار تشخیص می‌دهد

### افزودن کلیدهای ترجمهٔ جدید در کد

```tsx
// Use namespaced keys
const t = useTranslations("settings");
t("cacheSettings"); // maps to settings.cacheSettings in JSON

// Run check_translations.py to verify keys exist
python3 scripts/i18n/check_translations.py --verbose
```

### ملاحظات RTL

- عربی (`ar`) و عبرانی (`he`) localeهای RTL هستند
- از CSS سخت‌گیرانه `left`/`right` خودداری کنید — از ویژگی‌های منطقی `start`/`end` استفاده کنید
- QA بصری عدم تطابق‌های چیدمان RTL را از طریق `run-visual-qa.mjs` تشخیص می‌دهد

## مسائل شناخته‌شده و تاریخچه

### اصلاح `in.json` → `hi.json`

مولد در اصل برای هندی به‌جای کد صحیح ISO 639-1 `hi` از `code: "in"` (کد Google Translate منسوخ‌شده) استفاده می‌کرد. این امر یک `in.json` یتیم به‌عنوان کپی `hi.json` ایجاد کرد. با تغییر `code: "in"` به `code: "hi"` در `generate-multilang.mjs` و حذف فایل یتیم اصلاح شد.

> ⚠️ **ممیزی (2026-05-13):** پوشهٔ `docs/i18n/in/` هنوز روی دیسک وجود دارد (کپ کامل `hi/`). مولد ترجمه دیگر به آن نمی‌نویسد، اما درخت تاریخی هرس نشده است. پس از تأیید اینکه هیچ پیوند خارجی به مسیر قدیمی ارجاع نمی‌دهد، با `rm -rf docs/i18n/in/` بی‌خطر حذف می‌شود.

### `docs/i18n/README.md` خودکار تولید می‌شود

فایل `docs/i18n/README.md` به‌طور کامل توسط `generate-multilang.mjs docs` بازتولید می‌شود. هر ویرایش دستی از بین خواهد رفت. برای مستندات دست‌نویسی که باید ماندگار باشد از `docs/guides/I18N.md` (این فایل) استفاده کنید.

### فهرست خارجی کلیدهای غیرقابل‌ترجمه

allowlist `untranslatable-keys.json` از یک set پایتون inline در `validate_translation.py` به یک فایل JSON خارجی برای نگهداری آسان‌تر منتقل شد. اعتبارسنج‌کننده آن را در runtime بارگذاری می‌کند.

### اصلاح کد هندی در `generate-multilang.mjs`

مولد در اصل برای هندی به‌جای کد صحیح ISO 639-1 `hi` از `code: "in"` (کد Google Translate منسوخ‌شده) استفاده می‌کرد. این در upstream commit `952b0b22c` توسط `diegosouzapw` معرفی شد. با تغییر `code: "in"` به `code: "hi"` در آرایهٔ `LOCALE_SPECS` و حذف فایل یتیم `in.json` اصلاح شد.

### خروجی شمارش ignored در `validate_translation.py`

بررسی `quick` اکنون شمارش کلیدهای نادیده‌گرفته‌شده از `untranslatable-keys.json` را نمایش می‌دهد:

```
Missing: 0
Untranslated: 0
Ignored (UNTRANSLATABLE_KEYS): <varies per release>
```
