# موتورهای فشرده‌سازی

فشرده‌سازی RouteChi بر اساس قراردادهای موتور ساخته شده است. یک حالت می‌تواند یک موتور را مستقیماً
اجرا کند (`caveman` یا `rtk`) یا یک خط لوله ترکیبی قطعی که چندین موتور را به‌ترتیب اجرا می‌کند.

## حالت‌ها

| حالت        | مسیر موتور                         | ورودی مورد نظر                                |
| ----------- | ---------------------------------- | --------------------------------------------- |
| `off`       | هیچ‌کدام                            | حفظ دقیق پرامپت                              |
| `lite`      | راهبر‌های lite Caveman            | پاکسازی همیشه‌روشن با ریسک پایین              |
| `standard`  | Caveman                            | متراکم‌سازی پرامپت زبان طبیعی                 |
| `aggressive`| Caveman + خلاصه‌ساز‌های تاریخچه/ابزار | نشست‌های گفت‌وگوی طولانی                       |
| `ultra`     | Caveman + راهبر‌های هرس             | بازیابی محدودیت بافتار                        |
| `rtk`       | RTK                                | خروجی ترمینال، shell، build، test و git       |
| `stacked`   | خط لوله، پیش‌فرض `rtk -> caveman`  | لاگ‌های ابزار و متن ترکیبی، حداکثر صرفه‌جویی  |

## رجیستری موتور

رجیستری در `open-sse/services/compression/engines/registry.ts` قرار دارد. موتورها یک قرارداد
مشترک را افشا می‌کنند:

- `id`: شناسه پایدار موتور مانند `caveman` یا `rtk`
- `apply(text, config)`: مسیر اجرای قدیمی که توسط خطوط لوله ترکیبی استفاده می‌شود
- `compress(input, config)`: مسیر اجرای اصلی که متن + آمار را برمی‌گرداند
- `getConfigSchema()`: شکل شبیه JSON-Schema از پیکربندی معتبر را برمی‌گرداند
- `validateConfig(config)`: `{ valid, errors[] }` را برمی‌گرداند

ثبت با `registerCompressionEngine(engine)` (یا `registerEngine` برای موارد پیشرفته) انجام می‌شود،
که قبل از پذیرش، `assertValidEngine()` و `validateConfig(defaultConfig)` را فراخوانی می‌کند.
برای حذف یک موتور در زمان اجرا از `unregisterCompressionEngine(id)` استفاده کنید.

`strategySelector.ts` موتورهای داخلی را قبل از اجرای فشرده‌سازی ثبت می‌کند. این کار به پیش‌نمایش،
فشرده‌سازی زمان اجرا، حالت ترکیبی، آزمون‌ها و موتورهای آینده اجازه می‌دهد از همان مسیر اجرا استفاده کنند.

### فشرده‌سازی توضیحات MCP (مرتبط)

یک رجیستری جداگانه متادیتای توضیحات ابزار MCP را در سطح رجیستری فشرده می‌کند — به
`open-sse/mcp-server/descriptionCompressor.ts` و [MCP-SERVER.md](../frameworks/MCP-SERVER.md) مراجعه کنید. این
از قواعد Caveman استفاده مجدد می‌کند اما روی متادیتای ابزار عمل می‌کند، نه payloadهای درخواست.

### موتورهای داخلی اضافی

علاوه بر Caveman، RTK و LLMLingua-2، رجیستری چندین موتور تخصصی بدون‌تلفات / ساختاری را ارسال می‌کند
(توسط خطوط لوله ترکیبی، playground و آزمون‌ها استفاده می‌شوند):

| موتور        | شناسه          | کارکرد                                                                                                                                                                     |
| ------------ | -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| CCR          | `ccr`          | Content-Compress-Retrieve (H4): جایگزینی بلوک‌های بزرگ متنی پیوسته با ارجاعات مبتنی بر محتوا، تا بلوک‌های تکراری/بزرگ یک‌بار ارسال و سپس ارجاع داده شوند.                  |
| headroom     | `headroom`     | SmartCrusher (H3 + N5): فشرده‌سازی بدون‌تلفات جدولی payloadهای همگن آرایه JSON به فرمت ستونی `[N rows]`.                                                                     |
| ionizer      | `ionizer`      | نمونه‌گیری سطر head/middle/tail برای بلوک‌های همگن بسیار بزرگ، ذخیره middle حذف‌شده به عنوان ارجاع مبتنی بر محتوای CCR.                                                     |
| session-dedup| `session-dedup`| حذف تکراری مبتنی بر محتوا در طول نوبت‌ها (با الهام از TokenMizer): متن دیده‌شده در نوبت‌های قبلی همان نشست را حذف می‌کند.                                                  |

## Caveman

حالت Caveman بر متراکم‌سازی معنایی متن معمولی تمرکز دارد:

- بلوک‌های کد، URLها، JSON، مسیرها و داده‌های ساختاریافته را حفظ می‌کند
- حشو، تعارف، بافتار تکراری و عبارات پیوندی طولانی را حذف می‌کند
- از بسته‌های قواعد فایل آگاه به زبان در `open-sse/services/compression/rules/` پشتیبانی می‌کند
- از طریق حالت‌های قدیمی `standard`، `aggressive` و `ultra` در دسترس باقی می‌ماند

سطح داشبورد `Dashboard -> Context & Cache -> Caveman` است.

Caveman بالادست گزارش می‌دهد `~75%` توکن خروجی کمتر، `65%` میانگین صرفه‌جویی خروجی در بنچمارک‌ها
با محدوده `22-87%`، و یک ابزار فشرده‌سازی ورودی `~46%`. RouteChi هنگام مستندسازی صرفه‌جویی‌های
ترکیبی پرامپت/بافتار از عدد سمت ورودی Caveman استفاده می‌کند؛ حالت خروجی Caveman یک قابلیت
رفتار پاسخ جداگانه باقی می‌ماند.

## RTK

حالت RTK بر خروجی فرمان و ابزار تمرکز دارد:

- کلاس‌های خروجی مانند `git status`، `git branch`، `git diff`، Vitest/Jest/Pytest،
  آزمون‌های Cargo/Go، ساخت TypeScript/Vite/Webpack، ESLint، npm audit/installs، لاگ‌های Docker،
  `find`/`grep` در shell، ردپای پشته و لاگ‌های عمومی را تشخیص می‌دهد
- ۴۹ فیلتر JSON از `open-sse/services/compression/engines/rtk/filters/` را اعمال می‌کند
- از خط لوله اعلانی به سبک RTK پشتیبانی می‌کند: حذف ANSI، replace، میان‌بر match-output،
  حذف/نگهداری خطوط، برش هر خط، برش head/tail/max-line و بازگشت on-empty
- از فیلترهای پروژه با گیت اعتماد در `.rtk/filters.json` و فیلترهای سراسری در
  `DATA_DIR/rtk/filters.json` پشتیبانی می‌کند
- توالی‌های ANSI، نویز پیشرفت، خطوط تکراری و کدهای تکراری بی‌فایده را حذف می‌کند
- شکست‌های قابل اقدام، هشدارها، خلاصه‌ها، فایل‌های تغییر یافته و بافتار انتهایی را حفظ می‌کند
- می‌تواند به‌صورت اختیاری خروجی خام redact‌شده را برای بازیابی/اشکال‌زدایی از طریق مسیرهای مدیریت
  احراز هویت‌شده نگه دارد

سطح داشبورد `Dashboard -> Context & Cache -> RTK` است.

جزئیات عملیاتی برای فیلترهای سفارشی، اعتماد، verify و بازیابی خروجی خام در
[`RTK_COMPRESSION.md`](./RTK_COMPRESSION.md) آمده است.

RTK بالادست گزارش می‌دهد `60-90%` صرفه‌جویی برای فشرده‌سازی خروجی فرمان. مثال README آن نشان می‌دهد
یک نشست ۳۰ دقیقه‌ای Claude Code از `~118,000` توکن به `~23,900` می‌رسد، یعنی `79.7%` صرفه‌جویی.

## LLMLingua-2 (هرس معنایی)

حالت LLMLingua-2 **هرس معنایی توکن** را بر روی متن معمولی با استفاده از یک دسته‌بند توکن کوچک ONNX
انجام می‌دهد، که مکمل موتورهای مبتنی بر قاعده Caveman و RTK است:

- فقط متن معمولی در پیام‌های غیر سیستمی را فشرده می‌کند؛ بلوک‌های کد محصور شده و سایر ساختارهای
  حفاظت‌شده هرگز تغییر نمی‌کنند
- بک‌اند `@atjsh/llmlingua-2` (ONNX از طریق `@huggingface/transformers`) را در یک
  thread کارگر اجرا می‌کند، تا استنتاج مدل هرگز حلقه رویداد درخواست را مسدود نکند
- **قابل ترکیب است** (`stackPriority` 35): در یک خط لوله ترکیبی، پس از
  موتورهای ساختاری (CCR، session-dedup، headroom، Caveman) اما قبل از `ultra` اجرا می‌شود، زیرا
  هرس معنایی روی متن از قبل ساختاری فشرده‌شده مؤثرتر است — برای مثال
  `rtk -> caveman -> llmlingua`
- **در هر خطا fail-open می‌شود** (وابستگی‌های اختیاری مفقود، spawn کارگر، بارگذاری مدل، استنتاج
  یا timeout) → متن اصلی دست‌نخورده برمی‌گردد، هرگز خطا نمی‌دهد

محل موتور: `open-sse/services/compression/engines/llmlingua/`. سطح داشبورد
`Dashboard -> Context & Cache -> LLMLingua` است.

### مدل‌ها

مدل پیش‌فرض **TinyBERT** است (`atjsh/llmlingua-2-js-tinybert-meetingbank`، ~57 مگابایت،
سریع). یک مدل **BERT-base** با دقت بالاتر (`Arcoldd/llmlingua4j-bert-base-onnx`،
~710 مگابایت) از طریق فیلد `model` پیکربندی موتور در دسترس است. `@huggingface/transformers`
مدل انتخابی را به‌صورت تنبل از HuggingFace Hub در اولین فراخوانی به
`${DATA_DIR}/models/llmlingua` دانلود می‌کند (`modelStore.ts`)؛ یک بازنویسی پیکربندی `modelPath`
آن را به یک کپی محلی اشاره می‌دهد (نصب‌های آفلاین / ایزوله).

### وابستگی‌های اختیاری و نصب در صورت نیاز

استک همتای زمان اجرای LLMLingua که قابل هرس است، **اختیاری** است. سه بسته به‌عنوان
`optionalDependencies` در `package.json` اعلام شده‌اند و توسط بیلد تولید
(`scripts/build/prepublish.ts` آن‌ها را باندل نمی‌کند) **خارجی** نگه داشته می‌شوند:

| بسته                | نسخه (پین)   | یادداشت‌ها                                      |
| ------------------- | ------------ | ----------------------------------------------- |
| `@atjsh/llmlingua-2`| `2.0.3`      | بسته ورودی؛ سایرین را به عنوان همتا اعلام می‌کند |
| `@tensorflow/tfjs`  | `4.22.0`     | سنگین‌ترین وابستگی — حجم ~800 مگابایتی را تعیین می‌کند |
| `js-tiktoken`       | `^1.0.20`    | توکنایزر                                       |

`@huggingface/transformers` در نسخه `3.5.2` به‌عنوان یک وابستگی **اختیاری** پین شده است (مشترک با
مسیر embeddingهای محلی و همچنین در باندل مستقل نیز ردیابی می‌شود). اختیاری نگه‌داشتن آن از
توقف کل نصب RouteChi توسط شکست‌های postinstall ارائه‌دهنده CUDA در `onnxruntime-node` روی میزبان‌های CUDA 11
جلوگیری می‌کند؛ هنگامی که استک اختیاری غایب است، LLMLingua همچنان fail-open می‌شود. فقط سه
بسته بالا همتای SLM قابل هرس هستند. یک `npm install` استاندارد (توسعه) استک اختیاری را
به‌صورت خودکار نصب می‌کند مگر اینکه وابستگی‌های اختیاری حذف شده باشند.

**چرا در صورت نیاز:** بسته npm منتشر شده، باندل مستقل و image داکر
**بدون** این وابستگی‌ها ارسال می‌شوند تا سبک بمانند. هنگامی که غایب باشند، گیت وابستگی کارگر
(یک پروب resolve `@atjsh/llmlingua-2` در `worker.ts`) شکست می‌خورد و موتور
**به‌صورت بی‌صدا fail-open می‌شود** — انتخاب LLMLingua به یک no-op تبدیل می‌شود (متن دست‌نخورده برمی‌گردد، بدون
ثبت خطا). برای فعال‌سازی آن در یک محیط هرس‌شده، استک اختیاری را نصب کنید:

```bash
# پین کنید به نسخه‌های اعلام‌شده در optionalDependencies فایل package.json
npm install @atjsh/llmlingua-2@2.0.3 @tensorflow/tfjs@4.22.0 js-tiktoken
```

حدود **~800 مگابایت** کل: زمان اجرای TensorFlow.js + transformers غالب است؛
مدل TinyBERT ~57 مگابایت را در اولین استفاده اضافه می‌کند (نه از طریق npm).

بر اساس محیط:

- **توسعه / `npm install`** — به‌صورت خودکار نصب می‌شود مگر اینکه `--omit=optional` (یا `--no-optional`) داده باشید.
  نیازی به کار اضافی نیست.
- **npm سراسری (`npm i -g omniroute`) / مستقل** — دستور نصب بالا را در داخل
  پوشه بسته نصب‌شده اجرا کنید، یا بدون حذف وابستگی‌های اختیاری دوباره نصب کنید.
- **داکر** — دستور نصب را در یک لایه image مشتق‌شده اضافه کنید؛ image منتشر شده
  بر اساس طراحی سبک ارسال می‌شود.
- **VPS (PM2)** — در `node_modules` اپلیکیشن نصب کنید، سپس فرآیند را ری‌استارت کنید تا
  کارگر گیت را دوباره پروب کند.

**تأیید فعال بودن:** با انتخاب LLMLingua، متن معمولی واقعاً کوچک می‌شود (موتور
fail-open را متوقف می‌کند)، و اولین درخواست دانلود مدل را در
`${DATA_DIR}/models/llmlingua` راه‌اندازی می‌کند. گیت عمداً فقط `@atjsh/llmlingua-2` را پروب می‌کند —
سایر همتایان فقط ESM هستند و `require.resolve` حتی هنگام وجود روی آن‌ها خطا می‌دهد — بنابراین
کارگر همچنان fail-open می‌شود اگر هر همتایی در زمان `import()` واقعاً مفقود باشد.

## خطوط لوله ترکیبی

حالت ترکیبی مراحل خط لوله را به‌ترتیب اجرا می‌کند. پیش‌فرض این است:

```txt
rtk -> caveman
```

برای نشست‌های coding-agent که در آن پرامپت خروجی فرمان را با متن انسانی یا دستیار
ترکیب می‌کند، از این حالت استفاده کنید. RTK ابتدا لاگ‌های پرنویز ابزار را کاهش می‌دهد، سپس Caveman
متن زبان طبیعی باقی‌مانده را فشرده می‌کند.

مراحل خط لوله با `stackedPipeline` در تنظیمات فشرده‌سازی یا از طریق comboهای
فشرده‌سازی پیکربندی می‌شوند.

هنگامی که هر دو موتور همان payload مناسب را کاهش می‌دهند، صرفه‌جویی‌ها ترکیب می‌شوند:

```txt
combined = 1 - (1 - RTK savings) * (1 - Caveman input savings)
average  = 1 - (1 - 0.80) * (1 - 0.46) = 89.2%
range    = 1 - (1 - 0.60..0.90) * (1 - 0.46) = 78.4-94.6%
```

## فیلتر درخت دسترس‌پذیری MCP

فیلتر هوشمند درخت دسترس‌پذیری MCP یک لایه فشرده‌سازی پس از اجرا است که روی
**نتایج ابزار** MCP اجرا می‌شود، نه روی پرامپت‌ها یا بافتار. این فیلتر payloadهای پرحرف درخت
دسترس‌پذیری و snapshot مرورگر که توسط ابزارهایی مانند Playwright، computer-use و سرورهای MCP
اتوماسیون مرورگر برگردانده می‌شوند را هدف می‌گیرد.

### کارکرد

1. **حذف نویز** — مدخل‌های generic/text خالی را حذف می‌کند (`- generic:`، `- text: ""`)
2. **فروپاشی هم‌نیا** — هنگامی که ≥ `collapseThreshold` (پیش‌فرض 30) خط متوالی تکرار ساختاری
   باشند، آن‌ها را به اولین `collapseKeepHead` (پیش‌فرض 10) خط + یک خلاصه شمارش +
   آخرین `collapseKeepTail` (پیش‌فرض 5) خط فروپاشی می‌کند
3. **حفظ ref** — لنگرهای `[ref=eXX]` که توسط Playwright/computer-use نیاز هستند هرگز دست نمی‌خورند
4. **برش سخت** — اگر متن پس از فروپاشی همچنان از `maxTextChars` (پیش‌فرض 50,000) فراتر رود،
   با یک راهنمای ناوبری برش می‌خورد تا عامل بتواند به کار خود ادامه دهد

### محل موتور

```txt
open-sse/services/compression/engines/mcpAccessibility/
  index.ts            ← نقطه ورود smartFilterText()
  collapseRepeated.ts ← الگوریتم فروپاشی هم‌نیا
  constants.ts        ← DEFAULT_MCP_ACCESSIBILITY_CONFIG
```

### پیکربندی

توسط `compression.mcpAccessibility` در تنظیمات سراسری کنترل می‌شود (مهاجرت 056). پیکربندی پیش‌فرض:

```json
{
  "enabled": true,
  "maxTextChars": 50000,
  "collapseThreshold": 30,
  "collapseKeepHead": 10,
  "collapseKeepTail": 5,
  "minLengthToProcess": 2000
}
```

فیلتر فقط بر روی payloadهای نتیجه ابزار که `type` آن‌ها `"text"` است و طول آن‌ها از
`minLengthToProcess` فراتر می‌رود، اعمال می‌شود. این فیلتر بر فشرده‌سازی پرامپت یا
payloadهای درخواست تأثیری ندارد.

### صرفه‌جویی مورد انتظار

۶۰–۸۰٪ بر روی نتایج ابزار snapshot مرورگر، بسته به پیچیدگی صفحه. الگوریتم فروپاشی
O(n) در تعداد خط است و تأخیر ناچیزی اضافه می‌کند.

### این فیلتر در مقایسه با موتورهای فشرده‌سازی بالا

| جنبه        | Caveman / RTK / ترکیبی     | فیلتر دسترس‌پذیری MCP                  |
| ----------- | -------------------------- | -------------------------------------- |
| هدف         | پرامپت‌های درخواست / بافتار | نتایج ابزار MCP                        |
| راه‌انداز    | تنظیم حالت فشرده‌سازی      | `compression.mcpAccessibility.enabled` |
| محدوده       | همه پیام‌های SSE           | فقط نتایج ابزار                        |
| لنگرهای ref | N/A                        | به‌طور بدون قید و شرط حفظ می‌شوند       |

## comboهای فشرده‌سازی

comboهای فشرده‌سازی پروفایل‌های فشرده‌سازی نام‌گذاری‌شده‌ای هستند که می‌توانند به comboهای مسیریابی اختصاص یابند:

- `compression_combos`: حالت، خط لوله، پیکربندی RTK، پیکربندی زبان و نشانگر پیش‌فرض را ذخیره می‌کند
- `compression_combo_assignments`: یک combo فشرده‌سازی را به یک combo مسیریابی نگاشت می‌کند
- یکپارچه‌سازی زمان اجرا یک combo فشرده‌سازی اختصاص‌یافته را قبل از بازنویسی‌های عمومی combo حل می‌کند
- تحلیل‌ها شامل `compression_combo_id` و `engine` هستند

سطح داشبورد: `Dashboard -> Context & Cache -> Compression Combos`.

## سطح API

| مسیر                                  | هدف                                                             |
| ------------------------------------- | ---------------------------------------------------------------- |
| `/api/settings/compression`           | تنظیمات سراسری فشرده‌سازی (شامل پیکربندی `mcpAccessibility`)     |
| `/api/compression/preview`            | پیش‌نمایش هر حالت فشرده‌سازی                                     |
| `/api/compression/language-packs`     | فهرست بسته‌های زبانی Caveman موجود                               |
| `/api/context/caveman/config`         | نام مستعار تنظیمات Caveman                                       |
| `/api/context/rtk/config`             | پیش‌فرض‌ها و تنظیمات RTK                                         |
| `/api/context/rtk/filters`            | کاتالوگ فیلتر RTK                                                |
| `/api/context/rtk/test`               | اندپوینت پیش‌نمایش/آزمون RTK                                     |
| `/api/context/rtk/raw-output/[id]`    | بازیابی خروجی خام redact‌شده احراز هویت‌شده                      |
| `/api/context/combos`                 | CRUD combo فشرده‌سازی                                            |
| `/api/context/combos/[id]/assignments`| CRUD اختصاص combo مسیریابی                                       |
| `/api/context/analytics`              | نام مستعار تحلیل فشرده‌سازی                                      |

مسیرهای مدیریت نیازمند احراز هویت مدیریت یا بررسی‌های سیاست API-key هستند.

## ابزارهای MCP

فشرده‌سازی پنج ابزار MCP را افشا می‌کند:

| ابزار                                | محدوده              | هدف                              |
| ----------------------------------- | ------------------- | -------------------------------- |
| `omniroute_compression_status`      | `read:compression`  | تنظیمات، تحلیل‌ها، آمار کش       |
| `omniroute_compression_configure`   | `write:compression` | به‌روزرسانی تنظیمات سراسری       |
| `omniroute_set_compression_engine`  | `write:compression` | تنظیم حالت و خط لوله اختیاری     |
| `omniroute_list_compression_combos` | `read:compression`  | فهرست comboهای فشرده‌سازی        |
| `omniroute_compression_combo_stats` | `read:compression`  | خواندن تحلیل combo/موتور         |

## محدودیت‌های شناخته‌شده

- **LLMLingua-2 (SLM) نیازمند وابستگی‌های اختیاری هم‌مکان است.** کارگر فقط در یک
  بیلد تولید زمانی اجرا می‌شود که `@atjsh/llmlingua-2` + همتایان در
  `dist/node_modules` هم‌مکان شوند (به `scripts/build/colocateOptionals.mjs`، #4286 مراجعه کنید). بدون آن‌ها
  موتور fail-open می‌شود (متن اصلی را برمی‌گرداند). حل‌پازی کارگر دیگر به
  `import.meta.url` وابسته نیست (در باندل مستقل از بین می‌رود) — بر روی
  cwd زمان اجرا / `argv[1]` لنگر می‌اندازد.
- **بسته‌های زبانی Caveman `de` / `fr` / `ja` جزئی هستند.** این بسته‌ها قواعد `context` +
  `filler` + `structural` را ارسال می‌کنند اما بسته `dedup` / `ultra` ندارند، بنابراین شدت `ultra`
  برای آن زبان‌ها قوی‌تر از `full` نیست (آن‌ها فقط از قواعد خودشان استفاده می‌کنند — بازگشت بی‌صدا
  به قواعد `dedup`/`ultra` انگلیسی وجود ندارد، که متن خارجی را خراب می‌کرد).
  `en` / `es` / `id` / `pt-BR` کامل هستند. مشارکت `dedup.json` + `ultra.json`
  برای بسته‌های جزئی مورد استقبال قرار می‌گیرد.
- **تله‌متری ترکیبی فقط موتورهایی که فشرده کرده‌اند را فهرست می‌کند.** یک مرحله خط لوله ترکیبی که
  موتور آن اجرا شده اما ۰٪ صرفه‌جویی داشته، `stats:null` برمی‌گرداند و بنابراین در
  `engineBreakdown` ظاهر نمی‌شود — از یک مرحله که نادیده گرفته شده بود قابل تشخیص نیست. تشخیص
  "اجرا شده، ۰٪" از "نادیده گرفته شده" نیازمند تغییر مدل breakdown است و به تعویق افتاده است.
