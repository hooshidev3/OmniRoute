# 🗜️ راهنمای فشرده‌سازی پرامپت — OmniRoute

> به‌صورت خودکار ۱۵-۹۵٪ بر روی بافتار واجد شرایط صرفه‌جویی کنید. برای یک مرور سریع، به [بخش فشرده‌سازی README](../README.md#%EF%B8%8F-prompt-compression--save-15-95-eligible-tokens-automatically) مراجعه کنید.

## مرور کلی

OmniRoute یک خط لوله فشرده‌سازی پرامپت ماژولار را پیاده‌سازی می‌کند که **پیش‌گیرانه** قبل از رسیدن درخواست‌ها به
providerهای بالادست اجرا می‌شود. این یعنی صرفه‌جویی توکن شما به‌صورت شفاف اتفاق می‌افتد — نیازی به تغییر
در گردش کار شما نیست.

```
درخواست کلاینت
  → انتخابگر استراتژی فشرده‌سازی
    → بازنویسی combo؟ → از تنظیم combo استفاده کن
    → آستانه راه‌اندازی خودکار؟ → از حالت خودکار استفاده کن
    → حالت پیش‌فرض؟ → از تنظیم سراسری استفاده کن
    → خاموش؟ → فشرده‌سازی را رد کن
  → حالت فشرده‌سازی انتخاب‌شده
    → خاموش: بدون فشرده‌سازی
    → Lite: پاکسازی امن فاصله/قالب‌بندی (~15%)
    → Standard: حذف کلمات پرکن به سبک Caveman (~30%)
    → Aggressive: قدیمی‌سازی تاریخچه + خلاصه‌سازی (~50%)
    → Ultra: هرس اکتشافی + نازک‌سازی بلوک کد (~75%)
    → RTK: فیلتر کردن آگاه از فرمان خروجی ترمینال/ابزار (محدوده بالادستی 60-90%)
    → Stacked: خط لوله چند موتوری مرتب، معمولاً RTK سپس Caveman (محدوده واجد شرایط 78-95%)
  → درخواست فشرده‌شده → Provider
```

## حالت‌های فشرده‌سازی

### خاموش

هیچ فشرده‌سازی اعمال نمی‌شود. همه پیام‌ها بدون تغییر عبور می‌کنند.

### حالت Lite (~15% صرفه‌جویی، تأخیر <1ms)

امن‌ترین حالت — بدون تغییر معنایی، فقط پاکسازی قالب‌بندی:

| تکنیک                    | توضیحات                                    |
| ------------------------ | ------------------------------------------ |
| `collapseWhitespace`     | ادغام خطوط خالی متوالی و فاصله‌های انتهایی |
| `dedupSystemPrompt`      | حذف پیام‌های سیستم تکراری                  |
| `compressToolResults`    | فشرده‌سازی خروجی‌های پرحرف ابزار/تابع      |
| `removeRedundantContent` | حذف دستورالعمل‌های تکراری                  |
| `replaceImageUrls`       | کوتاه‌سازی URIهای داده تصویر base64        |

**بهترین برای:** استفاده همیشه‌روشن، گردش کارهای حیاتی از نظر ایمنی.

### حالت Standard (~30% صرفه‌جویی)

با الهام از [Caveman](https://github.com/JuliusBrussee/caveman) — کلمات پرکن و عبارات طولانی را
حفظ معنا حذف می‌کند:

- کلمات پرکن را حذف می‌کند ("please"، "I think"، "basically"، "actually")
- عبارات طولانی را متراکم می‌کند ("in order to" → "to"، "as a result of" → "because")
- تعارف مؤدبانه را حذف می‌کند ("Would you mind..."، "If you could possibly...")
- ۳۰+ قاعده regex تنظیم‌شده برای پرامپت‌های کدنویسی

**بهترین برای:** گردش کارهای روزانه کدنویسی، تیم‌های حساس به هزینه.

### حالت Aggressive (~50% صرفه‌جویی)

مدیریت هوشمند تاریخچه برای نشست‌های طولانی:

- **قدیمی‌سازی پیام** — پیام‌های قدیمی‌تر به‌تدریج فشرده می‌شوند
- **خلاصه‌سازی نتیجه ابزار** — خروجی‌های طولانی ابزار با خلاصه جایگزین می‌شوند
- **محافظ‌های یکپارچگی ساختاری** — اطمینان از اینکه جفت‌های `tool_use` + `tool_result` سازگار باقی بمانند
- **آگاهی از پنجره بافتار** — محدودیت‌های توکن هر مدل را رعایت می‌کند

**بهترین برای:** نشست‌های اشکال‌زدایی طولانی، کدبیس‌های بزرگ.

### حالت Ultra (~75% صرفه‌جویی)

حداکثر فشرده‌سازی برای سناریوهای حساس به توکن:

- **هرس اکتشافی** — پیام‌های زیر آستانه ارتباط را حذف می‌کند
- **نازک‌سازی بلوک کد** — مثال‌های کد تکراری را فشرده می‌کند
- **برش جستجوی دودویی** — نقطه برش بهینه برای پنجره بافتار را پیدا می‌کند
- همه قابلیت‌های حالت Aggressive شامل می‌شود

**بهترین برای:** زمانی که مرتباً به محدودیت‌های بافتار می‌رسید.

### حالت RTK (محدوده بالادستی 60-90%)

حالت RTK برای خروجی‌های پرحرف ابزار که در نشست‌های coding-agent ظاهر می‌شوند بهینه شده است:

- کلاس‌های فرمان/خروجی مانند `git status`، `git diff`، `git log`، test runnerها،
  ساخت TypeScript/Vite/Webpack، ESLint/Biome/Prettier، npm audit/installs، لاگ‌های Docker، خروجی
  infra و خروجی عمومی shell را تشخیص می‌دهد
- بسته‌های فیلتر JSON از `open-sse/services/compression/engines/rtk/filters/` را اعمال می‌کند
- ۴۹ فیلتر داخلی با نمونه‌های verify درون‌خطی ارسال می‌کند
- توالی‌های کنترل ANSI، نوارهای پیشرفت، خطوط تکراری و نویز غیر قابل اقدام را حذف می‌کند
- شکست‌ها، خطاها، هشدارها، فایل‌های تغییر یافته، خلاصه‌ها و انتهای خروجی طولانی را حفظ می‌کند
- از فیلترهای پروژه با گیت اعتماد، فیلترهای سراسری و بازیابی اختیاری خروجی خام redact‌شده پشتیبانی می‌کند

**بهترین برای:** نشست‌های عامل با transcripهای shell، build، test، git، grep و خروجی فایل.

### حالت Stacked (محدوده واجد شرایط 78-95%)

حالت Stacked چندین موتور فشرده‌سازی را به‌ترتیب قطعی اجرا می‌کند. خط لوله پیش‌فرض این است:

```txt
RTK -> Caveman
```

این ترتیب ابتدا خروجی ترمینال/ابزار را فشرده نگه می‌دارد، سپس متراکم‌سازی معنایی Caveman را بر روی
پرامپت زبان طبیعی باقی‌مانده اعمال می‌کند. خطوط لوله ترکیبی می‌توانند به‌صورت سراسری یا از طریق
comboهای فشرده‌سازی که به comboهای مسیریابی اختصاص یافته‌اند، پیکربندی شوند.

**بهترین برای:** بافتار ترکیبی با لاگ‌های بزرگ ابزار به علاوه دستورالعمل‌های انسانی یا خلاصه‌های دستیار.

## محاسبات صرفه‌جویی بالادست

OmniRoute صرفه‌جویی فشرده‌سازی را از دو منبع مستند می‌کند: بنچمارک‌های پروژه بالادست و
ترکیب موتور خود OmniRoute.

| منبع    | عدد README بالادست استفاده‌شده در اینجا                                                                         |
| ------- | --------------------------------------------------------------------------------------------------------------- |
| Caveman | توکن خروجی `~75%` کمتر، `65%` میانگین صرفه‌جویی خروجی بنچمارک، محدوده `22-87%`، و ابزار فشرده‌سازی ورودی `~46%` |
| RTK     | صرفه‌جویی خروجی فرمان `60-90%`؛ نشست نمونه `~118,000 -> ~23,900` توکن، یا `79.7%` صرفه‌جویی (`~80%`)            |

برای payloadهای همپوشان ابزار/بافتار، combo پیش‌فرض OmniRoute موتورها را ترکیب می‌کند:

```txt
RTK -> Caveman
```

صرفه‌جویی ترکیبی ضربی است، نه جمعی:

```txt
combined = 1 - (1 - RTK savings) * (1 - Caveman input savings)
average  = 1 - (1 - 0.80) * (1 - 0.46) = 89.2%
range    = 1 - (1 - 0.60..0.90) * (1 - 0.46) = 78.4-94.6%
```

این عدد `78-95%` زمانی اعمال می‌شود که هر دو RTK و Caveman بتوانند همان payload ورودی/بافتار را کاهش دهند.
حالت خروجی Caveman جداگانه است: هنگام فعال بودن، از صرفه‌جویی‌های خروجی خود Caveman (`65%`
میانگین، `~75%` عنوان، `22-87%` محدوده) استفاده کنید. کل صرفه‌جویی صورتحساب به ترکیب پرامپت/خروجی شما بستگی دارد.

## پیکربندی

### داشبورد

به `Dashboard → Context & Cache` بروید:

- **Caveman** — انتخاب حالت، بسته‌های زبانی، پیش‌نمایش و پیش‌فرض‌های سراسری
- **RTK** — پیش‌نمایش فیلتر فرمان، تنظیمات ایمنی RTK و کاتالوگ فیلتر
- **comboهای فشرده‌سازی** — خطوط لوله موتوری نام‌گذاری‌شده که به comboهای مسیریابی اختصاص یافته‌اند
- **آستانه راه‌اندازی خودکار** — فعال‌سازی خودکار فشرده‌سازی هنگام تجاوز تعداد توکن از آستانه

### بازنویسی هر combo

در `Dashboard → Context & Cache → Compression Combos`، یک combo فشرده‌سازی را به یک combo مسیریابی
اختصاص دهید:

```txt
Combo: "free-forever"
  Compression Combo: "coding-agent-stack"
  Pipeline: RTK -> Caveman
  Targets:
    1. if/kimi-k2-thinking
    2. qw/qwen3-coder-plus
```

این به شما اجازه می‌دهد از فشرده‌سازی ترکیبی روی providerهای رایگان/کدنویسی استفاده کنید در حالی که
حالت lite را روی اشتراک‌های پولی نگه دارید.

### بازنویسی هر درخواست

هدر درخواست `x-omniroute-compression` را ارسال کنید تا برنامه فشرده‌سازی برای یک درخواست
منفرد بازنویسی شود. این هدر بالاترین اولویت را دارد — از بازنویسی combo مسیریابی، پروفایل فعال،
راه‌اندازی خودکار و Default پنل سبق می‌گیرد. مقادیر ناشناخته نادیده گرفته می‌شوند (درخواست هرگز رد نمی‌شود) و
کلید اصلی سراسری همچنان همه‌چیز را گیت می‌کند: هنگامی که فشرده‌سازی به‌صورت سراسری خاموش است، هدر نمی‌تواند
آن را روشن کند. مقادیر:

| مقدار         | اثر                                                                                               |
| ------------- | ------------------------------------------------------------------------------------------------- |
| `off`         | بدون فشرده‌سازی برای این درخواست.                                                                 |
| `default`     | پروفایل Default مشتق‌شده از پنل (پروفایل فعال را نادیده می‌گیرد).                                 |
| `engine:<id>` | یک موتور منفرد هنگام فعال بودن، برای مثال `engine:rtk`.                                           |
| `<combo>`     | یک combo نام‌گذاری‌شده، ابتدا بر اساس نام (بدون حساسیت به حروف) سپس بر اساس id تطبیق داده می‌شود. |

برنامه اعمال‌شده در هدر پاسخ `X-OmniRoute-Compression: <mode>; source=<source>` بازتاب می‌یابد، که
در آن `<source>` یکی از `request-header`، `routing-override`، `active-profile`،
`auto-trigger`، `default` یا `off` است.

### API

```bash
# دریافت تنظیمات فشرده‌سازی
curl http://localhost:20128/api/settings/compression

# به‌روزرسانی تنظیمات فشرده‌سازی
curl -X PUT http://localhost:20128/api/settings/compression \
  -H "Content-Type: application/json" \
  -d '{"defaultMode":"stacked","autoTriggerMode":"stacked","autoTriggerTokens":32000}'

# پیش‌نمایش یک payload خاص RTK/stacked
curl -X POST http://localhost:20128/api/compression/preview \
  -H "Content-Type: application/json" \
  -d '{"mode":"rtk","messages":[{"role":"tool","content":"npm test output here"}]}'

# فهرست بسته‌های فیلتر RTK
curl http://localhost:20128/api/context/rtk/filters

# آزمون مستقیم RTK با متادیتای اختیاری فرمان
curl -X POST http://localhost:20128/api/context/rtk/test \
  -H "Content-Type: application/json" \
  -d '{"command":"npm test","text":"FAIL tests/example.test.ts\nError: boom"}'
```

## قدردانی‌ها

قواعد فشرده‌سازی حالت Standard با الهام از **[Caveman](https://github.com/JuliusBrussee/caveman)** اثر **[JuliusBrussee](https://github.com/JuliusBrussee)** (⭐ 51K+) است — پروژه ویروسی "چرا از توکن‌های زیاد استفاده کنیم وقتی توکن‌های کمی کار را انجام می‌دهند". Caveman گزارش می‌دهد `~75%` توکن خروجی کمتر، `65%` میانگین صرفه‌جویی خروجی بنچمارک، یک محدوده خروجی `22-87%` و یک ابزار فشرده‌سازی ورودی `~46%`.

حالت RTK با الهام از **[RTK - Rust Token Killer](https://github.com/rtk-ai/rtk)** اثر **[RTK AI](https://github.com/rtk-ai)** است — پروژه فشرده‌سازی خروجی فرمان با کارایی بالا برای فیلتر کردن ترمینال، build، test، git و خروجی ابزار. RTK گزارش می‌کند `60-90%` صرفه‌جویی، با نشست نمونه README که `~80%` صرفه‌جویی نشان می‌دهد.

## سیستم‌های پیشرفته فشرده‌سازی

علاوه بر ۷ حالت استاندارد، OmniRoute شامل چندین سیستم فشرده‌سازی پیشرفته است
که به‌صورت خودکار بر اساس بافتار کار می‌کنند.

### فشرده‌سازی آگاه از کش

برخی providerها (مانند Anthropic با prompt caching) از **prompt caching** پشتیبانی می‌کنند،
که به آن‌ها اجازه می‌دهد بخش‌هایی از پرامپت را برای کاهش هزینه و تأخیر کش کنند. هنگامی که
کش فعال است، فشرده‌سازی تهاجمی می‌تواند در واقع عملکرد را **مختل** کند
زیرا توکن‌های کش‌شده را تغییر می‌دهد و کش را نامعتبر می‌کند.

ماژول `cachingAware.ts` با **تشخیص بافتار کش** و
**تنظیم استراتژی فشرده‌سازی** متناسب با آن، این مشکل را حل می‌کند.

#### نحوه کار

1. **تشخیص بافتار کش** — بدنه درخواست را برای نشانگرهای `cache_control` اسکن می‌کند
2. **تشخیص providerهای کش‌کننده** — بررسی می‌کند آیا provider هدف از کش پشتیبانی می‌کند
3. **تنظیم استراتژی** — `aggressive`/`ultra` را به `standard` برای providerهای کش‌کننده تنزل می‌دهد
4. **رد کردن پرامپت سیستم** — پرامپت‌های سیستم معمولاً کش می‌شوند، بنابراین آن‌ها را فشرده نکنید
5. **استفاده از تحولات قطعی** — فقط از تحولاتی استفاده کنید که خروجی سازگار تولید می‌کنند

#### مثال کد

```ts
import {
  detectCachingContext,
  getCacheAwareStrategy,
} from "@omniroute/open-sse/services/compression/cachingAware";

const body = {
  model: "anthropic/claude-sonnet-4.5",
  messages: [{ role: "user", content: "Hello" }],
  cache_control: { type: "ephemeral" }, // ← نشانگر کش
};

const ctx = detectCachingContext(body, { provider: "anthropic" });
// → { hasCacheControl: true, provider: "anthropic", isCachingProvider: true }

const strategy = getCacheAwareStrategy("aggressive", ctx);
// → { strategy: "standard", skipSystemPrompt: true, deterministicOnly: true }
```

#### زمان استفاده

فشرده‌سازی آگاه از کش **همیشه روشن است** — نیازی به پیکربندی نیست. این فقط زمانی فعال می‌شود که:

- درخواست نشانگرهای `cache_control` داشته باشد
- provider هدف از prompt caching پشتیبانی کند (Anthropic، OpenAI و غیره)

### قدیمی‌سازی پیش‌رونده

مکالمات طولانی نوبت‌های پیام زیادی انباشته می‌کنند، اما نوبت‌های قدیمی‌تر کمتر
مرتبط می‌شوند. ماژول `progressiveAging.ts` **پیام‌ها را بر اساس فاصله نوبت تخریب می‌کند**:

- **نوبت‌های اخیر (0-3)**: کلمه‌به‌کلمه حفظ می‌شوند (جزئیات کامل)
- **نوبت‌های متوسط (4-8)**: فشرده‌سازی lite (فاصله، پاکسازی قالب‌بندی)
- **نوبت‌های قدیمی (9+)**: فشرده‌سازی Caveman (حذف حشو، خلاصه‌سازی)
- **نوبت‌های بسیار قدیمی (20+)**: به‌شدت خلاصه یا حذف می‌شوند

#### مثال کد

```ts
import { applyAging } from "@omniroute/open-sse/services/compression/progressiveAging";

const messages = [
  { role: "system", content: "You are a helpful assistant" },
  { role: "user", content: "What is 2+2?" },
  { role: "assistant", content: "4" },
  // ... 50 نوبت دیگر ...
];

const { messages: aged, saved } = applyAging(messages, {
  verbatim: 3, // اولین 3 نوبت: کلمه‌به‌کلمه
  light: 8, // نوبت‌های 4-8: فشرده‌سازی lite
  moderate: 20, // نوبت‌های 9-20: فشرده‌سازی caveman
  // نوبت‌های 21+: خلاصه‌سازی سنگین
});

// saved = تعداد توکن‌های ذخیره‌شده
```

#### زمان استفاده

قدیمی‌سازی پیش‌رونده برای حالت‌های `aggressive` و `ultra` **همیشه روشن است**. این به‌طور
خاص برای موارد زیر مؤثر است:

- نشست‌های کدنویسی طولانی‌مدت
- مکالمات چندروزه
- گردش کارهای agentic با فراخوانی‌های ابزار متعدد

### حالت خروجی Caveman

ماژول `outputMode.ts` **دستورالعمل‌های پرامپت سیستم** را تزریق می‌کند تا
مدل خودش خروجی فشرده و موجز تولید کند (سبک "caveman").

#### نحوه کار

به جای فشرده‌سازی ورودی، این حالت یک پرامپت سیستم مانند زیر اضافه می‌کند:

> "Reply in minimal words. Skip pleasantries. Use short sentences."

این به‌طور خاص برای موارد زیر خوب کار می‌کند:

- تولید کد (خروجی موجزتر = توکن کمتر)
- پرس‌وپاسخ سریع (نیازی به توضیحات مفصل نیست)
- پردازش دسته‌ای (حداکثر توان)

#### زمان استفاده

حالت خروجی Caveman **opt-in** است — آن را از طریق پیکربندی combo تنظیم کنید:

```json
{
  "strategy": "auto",
  "config": {
    "auto": {
      "outputMode": "caveman"
    }
  }
}
```

### فشرده‌سازی نتیجه ابزار

ماژول `toolResultCompressor.ts` **۵ استراتژی فشرده‌سازی تخصصی** را برای
نتایج ابزار فراهم می‌کند (فراخوانی تابع، خروجی‌های عامل، نتایج جستجو و غیره):

1. **فشرده‌سازی نتیجه جستجو** — نتایج تکراری را حذف می‌کند، top-N را نگه می‌دارد
2. **فشرده‌سازی خواندن فایل** — فایل‌های بزرگ را برش می‌دهد، header/importها را حفظ می‌کند
3. **فشرده‌سازی اجرای کد** — فقط stdout/stderr ضروری را نگه می‌دارد
4. **فشرده‌سازی query دیتابیس** — سطرها را محدود می‌کند، متادیتای پرحرف را حذف می‌کند
5. **فشرده‌سازی پاسخ API** — فیلدهای null را حذف می‌کند، آرایه‌ها را متراکم می‌کند

#### زمان استفاده

فشرده‌سازی نتیجه ابزار هنگام وجود فراخوانی‌های ابزار **همیشه روشن است**. نیازی به
پیکربندی نیست.

### خط لوله ترکیبی

حالت ترکیبی **چندین موتور را به‌ترتیب** اجرا می‌کند — معمولاً ابتدا RTK
(۶۰-۹۰٪ صرفه‌جویی بر خروجی ابزار)، سپس Caveman (۳۰٪ صرفه‌جویی اضافی بر روی
متن باقی‌مانده). این به **۷۸-۹۵٪ کل صرفه‌جویی** دست می‌یابد.

#### نحوه کار

```
ورودی (1000 توکن)
  → RTK (فیلتر آگاه از فرمان) → 200 توکن
    → Caveman (حذف حشو) → 140 توکن
  → خروجی (140 توکن، 86% صرفه‌جویی)
```

#### زمان استفاده

برای موارد زیر از حالت ترکیبی استفاده کنید:

- گردش کارهای سنگین از نظر ابزار (کدنویسی agentic، تحقیق)
- پردازش دسته‌ای حساس به هزینه
- زمانی که به حداکثر صرفه‌جویی توکن نیاز دارید

از طریق combo پیکربندی کنید:

```json
{
  "strategy": "auto",
  "config": {
    "auto": {
      "modePack": "stacked"
    }
  }
}
```

## بازنویسی‌های combo فشرده‌سازی

می‌توانید حالت فشرده‌سازی سراسری را **برای هر combo** بازنویسی کنید تا رفتار را برای
موارد استفاده متفاوت تنظیم کنید:

```json
{
  "id": "coding-combo",
  "strategy": "priority",
  "config": {
    "auto": {
      "weights": { "taskFit": 0.5 },
      "modePack": "quality-first"
    }
  },
  "compressionOverride": {
    "mode": "aggressive",
    "stackedPipelines": ["rtk", "caveman"],
    "preserveToolDefinitions": true
  }
}
```

این برای موارد زیر مفید است:

- **comboهای کدنویسی**: از حالت `aggressive` برای نشست‌های طولانی استفاده کنید
- **comboهای پرس‌وپاسخ سریع**: از حالت `lite` برای پاسخ‌های سریع استفاده کنید
- **comboهای سنگین از نظر ابزار**: از حالت `stacked` برای حداکثر صرفه‌جویی استفاده کنید
- **comboهای تولیدی**: از حالت `cache-aware` برای providerهای کش‌کننده استفاده کنید
