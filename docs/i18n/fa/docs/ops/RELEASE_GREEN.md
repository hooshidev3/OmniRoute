# Release-Green: نگه‌داشتن صف و شاخهٔ انتشار سبز

## مشکلی که حل می‌کند

**گیت کامل** (`.github/workflows/ci.yml` — shardهای واحد، vitest، ratchetها،
`package-artifact`، SonarQube، E2E) **فقط روی PR انتشار** اجرا می‌شود (PR → `main`). PRهایی که به
`release/**` هدف می‌گیرند فقط **fast-gates** را دریافت می‌کنند (`quality.yml`: تست‌های آسیب‌دیده TIA + typecheck +
lint). نتیجه: قرمزی‌ها بی‌صدا روی شاخهٔ انتشار انباشته می‌شوند و **در لایه‌هایی
از ~۴۰ دقیقه** در زمان انتشار، یکی‌یکی منفجر می‌شوند.

این «خانوادهٔ release-green» وجود دارد تا آن قرمزی‌ها را **پیش‌بینی** کند — معادل گیت کامل را **به‌صورت محلی /
خارج از انتشار**، در هر زمان تأیید کند، تا PR انتشار از اولین اجرای CI
سبز باشد.

> **اصل غیرقابل‌مذاکره:** هیچ‌کدام از این‌ها مشارکت‌کننده را مسدود نمی‌کند. ما یک بررسی الزامی اضافه نمی‌کنیم که
> PR او را شکست دهد. **drift** (ratchetها) برای rebaseline توسط نگه‌دارنده در انتشار است —
> هرگز دغدغهٔ مشارکت‌کننده نیست. هیچ قطعه‌ای **PR را نمی‌بندد** (سرقت اعتبار) و
> **تست را برای عبور تضعیف نمی‌کند**.

## خانواده (۴ قطعه) — و نحوهٔ اجرای مستقل هرکدام

| قطعه                                                                       | چیستی                                                                              | زمان اجرا                                                                          | اسکوپ                            |
| -------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- | -------------------------------- |
| **`/green-prs`** (راه‌حل A)                                                | اسکن به‌درخواست توسط نگه‌دارنده از **صف PRهای باز**                               | **مستقل، دوره‌ای** — و به‌ویژه **پیش از** یک `/generate-release`                    | کل صف PR → `release/**`          |
| **`/validate-release-green`** (راه‌حل C — `npm run check:release-green`)   | موتور اعتبارسنجی: گیت کامل را در برابر یک شاخه یا کاندیدای merge بازتولید می‌کند    | مستقل، در هر زمان                                                                   | یک شاخهٔ مشخص یا یک merge-PR       |
| **`/babysit <PR#>`**                                                       | **CI زنده** یک PR را تا سبز شدن هدایت می‌کند                                       | مستقل، به‌ازای PR                                                                   | یک PR منفرد                      |
| **`nightly-release-green.yml`** (راه‌حل D)                                  | workflow شبانهٔ خودکار؛ در قرمزی HARD یک issue باز می‌کند                          | خودکار (cron)                                                                      | شاخهٔ انتشار فعال                |

**پاسخ کوتاه به «آیا این فقط برای انتشارهاست؟»:** **خیر.** `/green-prs` طراحی شده
تا **دوره‌ای، بین انتشارها** اجرا شود. اجرای مستقل، استفادهٔ نرمال است — انتشار فقط
لحظه‌ای است که اجرای آن بیشترین ارزش می‌آفریند.

## راه‌حل C — `npm run check:release-green` (موتور)

اعتبارسنجی معادل انتشار را در برابر درخت کاری فعلی بازتولید می‌کند و هر قرمزی را طبقه‌بندی می‌کند:

- **HARD** (typecheck، خطاهای lint، unit، vitest، db-rules، public-creds،
  `package-artifact` اختیاری) → **نقص واقعی**؛ `exit 1`. روی شاخهٔ مبدأ اصلاح می‌شود (TDD، Rule #18).
- **DRIFT** (هشدارهای eslint، cognitive-complexity، file-size) → drift رatchet انباشته‌شده در
  چرخه، **تقصیر مشارکت‌کننده نیست**؛ فقط گزارش می‌شود و **توسط نگه‌دارنده در
  انتشار rebaseline می‌شود**. DRIFT **هرگز** کد خروج را تغییر نمی‌دهد — پس هرگز کسی را مسدود نمی‌کند.

```bash
npm run check:release-green                 # current branch (working tree)
node scripts/quality/validate-release-green.mjs --json   # structured output
node scripts/quality/validate-release-green.mjs --quick  # skips unit+vitest (drift+typecheck+lint only)
node scripts/quality/validate-release-green.mjs --with-build  # includes package-artifact (slow)
```

فقط تشخیص و **گزارش** (بدون auto-fix). هماهنگی fix-to-green در
`/green-prs` و `/review-prs` قرار دارد.

## راه‌حل A — `/green-prs` (اسکن صف)

رویه (خلاصه — برای جزئیات به مهارت `green-prs` مراجعه کنید):

۱. **فهرست‌کردن** صف PRهای باز در برابر شاخهٔ انتشار فعال.
۲. **triage** هر PR (قابل‌اجرا / قابل‌رد / نیازمند-نویسنده) — قابل‌رد/نیازمند-نویسنده
   **گزارش می‌شوند، بسته نمی‌شوند** (نویسنده تصمیم می‌گیرد).
۳. برای هر PR قابل‌اجرا، در یک **worktree ایزوله** (Rule #19)، PR را به نوک انتشار بیاورید و
   `npm run check:release-green` را اجرا کنید:
   - **HARD** → **روی شاخهٔ مشارکت‌کننده** ازطریق هم‌مؤلفگی اصلاح کنید (وضعیت «Merged» نویسنده را حفظ می‌کند)؛
     دوباره اجرا کنید تا همهٔ HARDها پاک شوند.
   - **DRIFT** → رهایش کنید؛ در انتشار rebaseline می‌شود.
۴. **گزارش** یک جدول PR × (رأی، قرمزی‌های HARD، اصلاح‌شده؟، DRIFT، اکنون release-green؟).

می‌تواند صف را بدون merge **آماده** کند؛ فقط هنگام درخواست صریح merge می‌کند — و هرگز PR را نمی‌بندد.

## ضرباهنگ توصیه‌شده

- **`/green-prs` را به‌صورت دوره‌ای** (مثلاً هفتگی) و **همیشه پیش از یک
  `/generate-release`** اجرا کنید.
- **`nightly-release-green.yml`** (راه‌حل D) را به‌عنوان سیگنال پیوسته نگه دارید: وقتی یک
  issue قرمزی HARD باز می‌کند، زمان اسکن است.
- از **`/validate-release-green`** به‌صورت ad-hoc برای بررسی یک شاخه یا یک کاندیدای merge مشخص استفاده کنید.
- از **`/babysit <PR#>`** هنگامی که یک PR مشخص باید روی CI زنده به سبز هدایت شود استفاده کنید.

## رابطه با انتشار

- `/generate-release` در **فاز ۰ (pre-flight)** اعتبارسنجی را فراخوانی می‌کند: DRIFT را rebaseline می‌کند و
  HARD را پیش از باز کردن PR انتشار اصلاح می‌کند.
- `/review-prs` از گیت release-green در مرحلهٔ تصمیم merge استفاده می‌کند (سبز-قبل‌از-merge).

هدف همهٔ قطعات یکسان است: **یک PR انتشار سبز در اولین اجرای CI**، به‌جای موج‌سواری روی
قرمزی‌ها در لایه‌های ۴۰ دقیقه‌ای در روز انتشار.
