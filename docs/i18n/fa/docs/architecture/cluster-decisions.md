# تصمیمات خوشه‌ای — پروفایل‌های اختیاری سایدکار

**وضعیت:** پیشنهاد (در انتظار بازبینی @diegosouzapw)
**تاریخ:** 2026-06-20
**مراجع:** [#3932](https://github.com/borhandarabi/routechi/issues/3932)، PR #4381

## خلاصه

دو پروفایل compose اختیاری (`memory` و `bifrost`) برای استقرار ۸-سرویسی موجود در [`docker-compose.yml`](../../docker-compose.yml). رفتار پیش‌فرض هنگام راه‌اندازی **بدون تغییر** باقی می‌ماند: ۳ نمونه `omniroute` + Caddy + Redis + CliproxyAPI. این دو پروفایل جدید Qdrant و Bifrost را به‌عنوان سایدکارهای اختیاری اضافه می‌کنند که با `docker compose --profile <name> up` فعال می‌شوند. **هیچ سرویس موجودی حذف یا جایگزین نمی‌شود.**

## چرا این رویکرد محافظه‌کارانه است

شکل فعلی استقرار RouteChi از قبل سبک و اثبات‌شده است:

- **`redis:7-alpine`** بار کاری محدودیت نرخ/حافظه نهان را در مقیاس تولید مدیریت می‌کند.
- **SQLite + sqlite-vec + FTS5** حافظه محلی + برداری + جست‌وجوی متنی را پوشش می‌دهند (به [`src/lib/memory/vectorStore.ts:108`](../../src/lib/memory/vectorStore.ts) مراجعه کنید).
- **Caddy** از قبل متعادل‌کننده بار (LB) + پایان‌دهنده TLS است ([`docker-compose.yml`](../../docker-compose.yml)).
- **Bifrost** از قبل به‌عنوان مسیریاب Tier-1 در [`src/app/api/v1/relay/chat/completions/bifrost/route.ts`](../../src/app/api/v1/relay/chat/completions/bifrost/route.ts) ادغام شده است (پروکسی سایدکار با سوئیچ قطع از طریق متغیر محیطی `BIFROST_ENABLED` — برای دور زدن سایدکار و عبور به مسیر TS مقدار `=0` تنظیم کنید).

این دو پروفایل در اینجا **گزینه‌های مقیاس‌دهی افقی برای استقرارهایی هستند که به سقف SQLite می‌رسند** — نه مهاجرت. هر دو به‌صورت پیش‌فرض خاموش هستند.

## دو پروفایل

### `memory` — سایدکار حافظه برداری Qdrant

**چه زمانی فعال کنید:**

- بیش از ۱ میلیون embedding به ازای هر استقرار (sqlite-vec در مقیاس بالا کند می‌شود).
- استقرار چندنمونه‌ای که به حالت برداری مشترک بین `omniroute-1/2/3` نیاز دارد.
- در حال حاضر یک خوشه Qdrant خارجی دارید (Qdrant Cloud یا روی پیشم).

**چه چیزی اضافه می‌کند:**

| سرویس    | تصویر                  | پورت‌ها      | یادداشت‌ها                                              |
| -------- | ---------------------- | ------------ | ----------------------------------------------------- |
| `qdrant` | `qdrant/qdrant:v1.12.4` | `6333` HTTP | شاخص HNSW؛ حجم پایای `omniroute_qdrant_data`           |

**فعال‌سازی:** مقدار `qdrantEnabled = true` را در رابط تنظیمات تغییر دهید **یا** متغیر محیطی `QDRANT_HOST=qdrant` را تنظیم کنید. برای قواعد اولویت به [`src/lib/memory/qdrant.ts:60`](../../src/lib/memory/qdrant.ts) مراجعه کنید (جدول تنظیمات ← متغیر محیطی ← پیش‌فرض).

**متغیرهای محیطی:** `QDRANT_HOST`، `QDRANT_PORT`، `QDRANT_API_KEY`، `QDRANT_COLLECTION`، `QDRANT_VECTOR_SIZE`، `QDRANT_HNSW_EF_CONSTRUCT` (به خطوط ۱۶۷۲-۱۶۸۳ در `.env.example` مراجعه کنید).

### `bifrost` — سایدکار مسیریاب Tier-1 بیفروست

**چه زمانی فعال کنید:**

- حداقل ۳ نمونه `omniroute` اجرا می‌کنید و می‌خواهید چرخش provider در یک فرآیند Go واحد متمرکز شود.
- یک سطح واحد برای ممیزی/ثبت گزارش درخواست‌های provider بالادستی در میان همه نمونه‌ها می‌خواهید.
- مقیاس‌دهی افقی لایه مسیریابی Tier-1 را مستقل از نمونه‌های RouteChi می‌خواهید.

**چه چیزی اضافه می‌کند:**

| سرویس     | تصویر                             | پورت‌ها | یادداشت‌ها                                                          |
| --------- | --------------------------------- | ------- | ------------------------------------------------------------------ |
| `bifrost` | `ghcr.io/maximhq/bifrost:1.5.21`  | `8080`  | مسیریاب Tier-1 مبتنی بر Go؛ حجم پایای گزارش‌ها `omniroute_bifrost_logs` |

**فعال‌سازی:** مقدار `BIFROST_BASE_URL=http://bifrost:8080` را در `.env.example` تنظیم کنید. مسیر پروکسی سایدکار موجود در [`src/app/api/v1/relay/chat/completions/bifrost/route.ts`](../../src/app/api/v1/relay/chat/completions/bifrost/route.ts) (اضافه‌شده در PR #4381) این را به‌صورت خودکار برمی‌دارد.

**متغیرهای محیطی:** `BIFROST_BASE_URL`، `BIFROST_API_KEY`، `BIFROST_STREAMING_ENABLED`، `BIFROST_TIMEOUT_MS` (به خطوط ۱۶۸۵-۱۶۹۵ در `.env.example` مراجعه کنید).

## کارهایی که این PR به‌صراحت انجام نمی‌دهد

موضوع اصلی issue اولیه بازنویسی بزرگتر خوشه بود. پس از ممیزی شکل واقعی بار کاری، موارد زیر به دلایل ذکرشده **رد شده‌اند**:

| مؤلفه                              | رأی     | دلیل                                                                                                       |
| ---------------------------------- | ------- | -------------------------------------------------------------------------------------------------------- |
| **Dragonfly**                      | **حذف** | `redis:7-alpine` برای بار کاری محدودیت نرخ در مقیاس تولید مناسب است؛ سقفی برای شکستن وجود ندارد.            |
| **NATS**                           | **حذف** | هر نمونه `omniroute` یک فرآیند Node.js واحد است؛ بار کاری pub/sub چندفرآیندی وجود ندارد.                  |
| **PostgreSQL**                     | **حذف** | SQLite + sqlite-vec + FTS5 هر ۳ مورد استفاده را پوشش می‌دهند؛ ۹۷ مهاجرت + بسته‌بندی Electron مانع مهاجرت می‌شوند. |
| **Neo4j**                          | **حذف** | مسیریابی یک join ۵-جدولی است؛ CTE بازگشتی روی SQLite کافی است.                                          |
| **MinIO**                          | **حذف** | بار کاری blob چندمگابایتی وجود ندارد؛ تصاویر/صدا پروکسی‌های عبوری هستند.                                  |
| **pgvector / pg_ai / pg_textsearch** | **حذف** | همان دلیل سقف SQLite مانند PostgreSQL؛ اکوسیستم pgvector پراکنده است.                                  |
| **HAProxy / Envoy**                | **حذف** | Caddy از قبل LB + TLS را انجام می‌دهد؛ هر دو به‌صراحت به‌عنوان مسیریاب Tier-1 رد شده‌اند (به `AGENTS.md` مراجعه کنید). |

اگر مورد استفاده آینده یکی از این‌ها را اثبات کند، این سند مکان مناسب برای اصلاح است.

## استقرار ۴ هفته‌ای (در صورت تأیید)

1. **هفته ۱** — ادغام این PR + راستی‌آزمایی پروفایل‌های اختیاری با یک استک compose با ۳ نمونه.
2. **هفته ۲** — فعال‌سازی کامل Bifrost برای OpenAI/Claude/Gemini/Ollama (۴ از ۱۴+ provider) با استفاده از مسیر پروکسی سایدکار در [`src/app/api/v1/relay/chat/completions/bifrost/route.ts`](../../src/app/api/v1/relay/chat/completions/bifrost/route.ts) (با `BIFROST_ENABLED` کنترل می‌شود، در زمان اجرا قابل قطع است).
3. **هفته ۳** — پروفایل حافظه Qdrant در یک استقرار آزمایشی فعال شد؛ اندازه‌گیری تفاضل تأخیر در مقابل sqlite-vec.
4. **هفته ۴** — بررسی‌های سلامت قابلیت مشاهده‌پذیری (کدهای خروج `docker compose ps` + تست‌های دودی `wget`)؛ بازخوانی ۷۱-ستونی مطابق ADR-041.
